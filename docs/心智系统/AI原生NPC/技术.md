# AI NPC — 技术可行性

## LLM Provider 抽象层

云端 API 为主路径，本地模型为备降。架构上 LLM 调用接口统一抽象，两种实现透明切换：

```
NPC 发起 LLM 调用 → LLMProvider (抽象接口)
                      ├─ CloudProvider → 鉴权服务器获取临时 token → 厂商 API（主路径）
                      └─ LocalProvider → 本地 Ollama/llama.cpp（无网络降级）
```

- **CloudProvider**（默认，订阅玩家）：不直接携带 API key。先调鉴权服务器验证订阅 → 换取临时 token → 调用厂商 API。临时 token 有过期时间，泄露无害
- **LocalProvider**：无网时自动降级。MVP 阶段不做深度优化——它只是安全网

模型分层策略：
```
行为决策 → 轻量模型（1-3B，云端/本地） → < 0.5秒
  对话    → 标准模型（7B+，云端优选）    → 1-3秒（对话中可接受）
  反思    → 标准模型，后台异步执行        → 不阻塞
```

## 整体架构：Godot + Python 进程分离

```
┌──────────────────────┐     TCP/localhost     ┌──────────────────────┐
│      Godot 4.x       │ ◄──────────────────► │   Python 后端进程    │
│  UI / 场景 / 渲染    │    MessagePack        │                      │
│  ┌────────────────┐  │                       │  基因系统 / NPC AI   │
│  │ 薄渲染层        │  │                       │  世界模拟 / 存档     │
│  │ 发送事件/请求    │  │                       │  LLM 接入            │
│  └────────────────┘  │                       │  Mod 脚本 = Python   │
└──────────────────────┘                       └──────────────────────┘
```

- **Godot**：仅负责渲染、UI、输入、音频。不包含游戏逻辑。
- **Python**：承载全部核心逻辑——基因系统、NPC AI、世界模拟、存档、Mod 加载。
- **通信**：MessagePack 序列化，TCP localhost。事件和状态更新通过统一事件总线传递。
- **选择此方案的原因**：
  - Python 作为 mod 脚本，零额外语言学习成本，社区基数最大
  - LLM 接入自然（openai/llama-cpp-python 等生态）
  - 渲染与逻辑彻底分离，独立开发、测试、调试
  - 未来换渲染层不影响核心逻辑

## 事件驱动通信

游戏内所有可被干预的节点暴露为事件。Godot 负责将玩家输入转换为事件发送给 Python，Python 处理事件并返回状态变更。

事件流：
```
玩家操作 → Godot 捕获 → 事件消息 → Python 事件总线 → 各模块处理 → 状态变更 → Godot 渲染更新
                                                                         → Mod 脚本响应
```

核心事件示例：
```
on_creature_captured    on_gene_extracted     on_npc_decision
on_breeding_complete    on_rule_set           on_archive_written
on_consciousness_transfer   on_death   on_building_constructed
```

世界状态同步采用**差异推送**：Python 只向 Godot 发送当前视野内发生变化的对象状态，而非全量。

## Mod 支持：三层基础设施

目标：修改深度 = Minecraft 级别，创作门槛 = RimWorld 级别。

### 第 1 层：数据驱动（所有内容外置）

从 MVP 第一天起，以下内容全部为 JSON 数据文件，不硬编码：

- 基因模板、物种定义、建筑配方、物品属性
- 科技树节点、地图生成参数、NPC 行为权重
- 本地化文本

Mod 加载机制：`mods/<mod名>/` 下的 JSON 文件覆盖 `data/` 默认值。加载顺序 = 优先级。

### 第 2 层：事件钩子

每个核心事件预留钩子，Mod 在 manifest 中声明监听的信号，Python 事件总线在事件触发时调用 Mod 注册的处理函数。

### 第 3 层：Python 脚本 = Mod

不做 DSL，不嵌入 Lua。Modder 直接写 Python。一份 `.py` 文件丢进 `mods/` 目录即生效。

Mod manifest 格式（`mod.json`）：
```json
{
  "name": "bioluminescent_mushrooms",
  "version": "1.2.0",
  "author": "...",
  "description": "添加会发光的蘑菇物种",
  "depends": [],
  "conflicts": [],
  "hooks": ["on_creature_spawned", "on_night_tick"]
}
```

### 社区基础设施规划

- **Steam Workshop**：目标平台，一键订阅和自动更新
- **Modpack 支持**：manifest 中 `depends`/`conflicts` 字段支持声明式依赖，多 mod 可组合
- **服务端/客户端区分**：将来多人模式下，Mod 标记为 `client` / `server` / `both`

## Cython 渐进优化

核心原则：**MVP 阶段纯 Python 跑通，热点出现时才用 Cython 加速。**

Cython 是 Python 的超集——在 Python 代码中加 C 类型声明，编译为 `.so` 原生模块。调用方无感知（`import` 的是 `.py` 还是 `.so` 完全透明）。

候选热点模块：

| 模块 | Python 可行？ | 需要 Cython？ | 原因 |
|---|---|---|---|
| 基因重组计算 | 多数情况够 | 后期大规模育种时 | 纯数值，循环密集 |
| NPC 决策 | 够 | 一般不需要 | Utility AI 是查表 + 权重加和 |
| 视野外记录生成 | 够 | 一般不需要 | 概率采样，numpy 已足够 |
| 存档序列化 | 不够 | 首选 | 大量对象序列化是纯 CPU |
| 地图生成 | 可能不够 | 候选 | Perlin 噪声等算法密集 |

引入策略：
1. **MVP 阶段**：全模块纯 Python，验收标准仍是 FPS >60 @20 NPC（Python + numpy 足够）
2. **热点出现时**：将对应模块独立的 `.py` 文件复制为 `.pyx`，加类型声明，编译
3. **调用方不变**：`from module import func` 不管底层是 Python 还是 Cython

不用 Rust/PyO3 的原因：需额外学习成本。Cython = Python 语法 + C 类型，你熟悉 C，掌握成本远低于 Rust。性能天花板接近 C，足够覆盖模拟经营类游戏的全部计算需求。
