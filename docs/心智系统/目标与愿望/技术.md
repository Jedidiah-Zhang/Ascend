## 优先级自动计算

非 LLM 生成的目标，priority 由动机层根据来源需求自动计算：

```
priority = clamp(urgency × source_weight, 0, 1)

source_weight 基准：
  生理需求（饱腹/水分）:  1.0
  生理需求（体温/健康）:  0.95
  生理需求（排泄/体力）:  0.85
  心理满足度:            0.6
  角色维护:              0.5
  LLM 设定:              使用 LLM 指定的 priority，不自动计算
```

LLM 设定的目标使用 LLM 指定的 priority，不受动机层自动覆盖。

## 排序算法

```
def sort_goals(goals):
    deadline_order = {
        "immediate": 0, "today": 1, "this_week": 2,
        "this_season": 3, "indefinite": 4
    }
    return sorted(goals, key=lambda g: (
        deadline_order[g.deadline],
        -g.priority,
        -g.frustration
    ))
```

排序结果中第一个为当前目标，其余按需进入待办列表。

## 目标切换逻辑

```
def select_current_goal(goals, current_goal):
    sorted_goals = sort_goals(goals)
    top = sorted_goals[0]
    
    if top.id == current_goal.id:
        return current_goal  # 不变
    
    if top.deadline == "immediate" and current_goal.deadline != "immediate":
        # 紧急中断：当前目标挂起
        current_goal.status = "suspended"
        pending_goals.insert(0, current_goal)
    
    return top
```

非紧急的切换（新目标 deadline 不比当前更紧迫）只在当前目标完成或被放弃时发生。

## 目标完成处理

```
def on_goal_completed(goal_id):
    goal = get_goal(goal_id)
    goal.progress = 1.0
    
    role = match_role_from_goal(goal.label)
    if role:
        add_role(npc, role)
        generate_role_maintenance_goals(npc, role)
    
    remove_goal(npc, goal_id)
    promote_from_pending(npc)
```

`match_role_from_goal` 是一个简单的关键词匹配表，将目标 label 映射到角色类型：

| 目标关键词 | 创建角色 |
|---|---|
| 开.*酒馆/店铺/作坊 | 店主 |
| 成为.*首领/族长 | 族群首领 |
| 建立.*家庭 | 家长 |

LLM 在设定目标时也可以在 metadata 中显式指定完成后应创建的角色。

## prerequisites 变更监控

系统在每个游戏天结束时扫描所有非 immediate 目标的 prerequisites：

```
for goal in all_non_immediate_goals(npc):
    for prereq in goal.prerequisites:
        if prereq_permanently_lost(prereq, npc):
            goal.pending_review = True
            break
```

`prereq_permanently_lost` 的判断逻辑：
- `has_skill`：该技能不在 NPC 的先天/习得/角色技能中，且无法在当前条件下习得（无教师、无学习材料）
- `has_item`：该物品已销毁且无法在当前条件下重新获取
- `near`：该位置类型在可到达范围内不存在

标记为 `pending_review` 的目标在下次 LLM 反思时一并处理，不单独唤醒 LLM。

## 角色子目标生成

角色维护子目标由角色定义中的生成规则 + LLM 生成组成：

```
# 系统自动检测的条件→目标
role_maintenance_rules = {
    "酒馆老板": [
        {condition: "inventory.麦芽 < threshold", goal: "补充麦芽库存", deadline: "today"},
        {condition: "customer_count > service_capacity * 1.5", goal: "招一个帮手", deadline: "this_week"},
    ]
}

# LLM 首次获得角色时生成个性化子目标
on_role_acquired(npc, role):
    prompt = f"你成为了{role}。除了日常经营，你有哪些个人想达成的目标？"
    llm_generated_subgoals = call_llm(prompt)
```

系统规则覆盖常规维护（确保 NPC 不失能），LLM 生成的子目标提供个性化（"我想研发一种加蜂蜜的新酒"）。

## 冲突检测

在将目标送入执行层前，检测当前目标与紧急需求目标是否冲突：

```
def check_conflict(goal_a, goal_b):
    # 冲突条件：两个目标都需要NPC在场，但位置不同且时间重叠
    if goal_a.requires_presence and goal_b.requires_presence:
        if goal_a.location != goal_b.location:
            if goal_a.deadline == goal_b.deadline == "immediate":
                return True
    return False
```

冲突时，两个目标打包送入 LLM 裁决。裁决结果直接驱动执行层，不需要额外仲裁逻辑。
