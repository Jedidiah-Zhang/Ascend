# 世界树 — 技术

世界树（`ascend/world_tree/`）是连接所有模块的骨干。模块 A 发布事件时不关心模块 B 是否在监听，订阅者按 event_type 匹配接收。

## 模块结构

```
ascend/world_tree/
├── __init__.py          # 模块级单例 world_tree，公共 API 导出
├── tree.py              # WorldTree 类（发布/订阅/查询/trim/configure/warmup）
├── event.py             # Event dataclass
├── affected.py          # AffectedParty dataclass
├── graph.py             # EventGraph 事件关系有向图
├── archive.py           # EventArchive SQLite 归档
├── schema.py            # EventSchema 单事件类型字段规范
├── registry.py          # SchemaRegistry 事件类型注册表
└── schema.sqlite.sql    # 归档 DDL（events、event_entities、event_edges）
```

## 事件数据结构

```
Event {
  id: UUID hex (自动生成),
  timestamp: float (游戏秒),
  location: (chunk_x, chunk_y, tile_x?, tile_y?),

  initiator_type: "system" | "npc" | "player",
  initiator_id: string,
  affected: [AffectedParty],     // {entity_id, role: "subject"|"witness"|"recipient"}

  event_type: string,            // "weather_change" "npc_action" "observation" ...
  weight: int (默认 1),           // trim 优先级，1=普通 3=重要 5=关键
  data: dict[str, Any],          // 事件类型特定的内容，JSON 可序列化

  // 事件图关系
  caused_by: [EventID],          // 因果上游
  observes: EventID | None,      // 观测引用（仅观测事件）
  co_participants: [EntityID],   // 共享参与方
}
```

## 事件类型注册

每个产生事件的模块在模块级注册自己的事件类型 schema，紧挨发布代码旁边：

```python
# clock.py — 时间系统注册自己的事件
world_tree.register_event_schema(
    "game_minute",
    required={"step": (int, float), "mode": str, "tick_count": int, "game_time": (int, float)},
    description="每 tick 发布一次",
)
```

```python
# manager.py — 实体系统注册自己的事件
world_tree.register_event_schema(
    "entity_spawned",
    required={"entity_id": str, "entity_type": str, "position": tuple},
    description="新实体生成",
)
```

- 谁发布谁注册，schema 定义紧挨发布代码
- 未注册的事件类型在 publish 时静默跳过校验
- `world_tree.schema_registry.list_schemas()` 可查看所有已注册类型

## 世界树核心设计

### WorldTree 类

`WorldTree` 是事件系统的核心类。模块级单例 `world_tree` 通过 `from ascend.world_tree import world_tree` 获取。

```
WorldTree
├── _event_log: list[Event]               热数据（内存）
├── _id_index: dict[str, Event]           ID → Event O(1) 查找
├── _entity_index: dict[str, list[Event]] 实体索引（按时间有序，二分定位）
├── _spatial_index: dict[tuple, set[Event]] 空间索引（chunk 分桶）
├── _graph: EventGraph                    因果有向图
├── _archive: EventArchive | None         冷数据（SQLite，含边表）
├── _schema_registry: SchemaRegistry | None 事件 schema 校验（可选）
├── _lock: threading.RLock                线程安全
├── _publish_count / _trim_count          运行统计
└── _async_executor: ThreadPoolExecutor   subscribe_async 线程池
```

### 发布流程

```text
publish(event):
    锁外: _validate_event(event)             校验必填字段 + initiator
          _validate_schema(event)            校验 data 字段（仅已注册类型）
    锁内: append → _event_log                O(1)
          更新 _id_index                     O(1)
          更新 _entity_index                 O(A) A=受影响方数
          更新 _spatial_index                O(1)
          _graph.add_event                   O(R) R=关系边数
          抓取订阅者快照
    锁外: _dispatch(event, callbacks)
    自动 trim: len > max_memory_events → trim 到 max//2
            增量清理索引（O(T)，T=被移除事件数）
```

### 分发通道

| 方法 | 执行线程 | 阻塞 publish | 适用场景 |
|------|---------|-------------|---------|
| `subscribe` | publish 调用线程 | 是 | 内存操作（状态更新、计数） |
| `subscribe_async` | 线程池 | 否 | 文件 I/O、网络 I/O |

`subscribe` 返回一个 `unsubscribe()` 函数。`"*"` 通配所有事件类型。

### 线程安全

- 所有公共读写方法由 `threading.RLock` 保护
- `publish()` 在锁内完成写入和订阅者快照，锁外执行回调
- 回调异常被 `try/except` 隔离，单个回调失败不影响其他订阅者
- 查询方法（`get_events_in_range` 等）持锁读取，确保一致性
- 归档查询在锁外执行（SQLite 有独立的并发控制），不占总线锁

### 事件校验

两层校验，在锁外执行，快速失败：

**基本校验**（`validate=True` 时启用）：
- `event_type` 非空白
- `initiator_id` 非空白
- `initiator_type` ∈ {"system", "npc", "player"}
- `timestamp` 非负
- `location` 为至少 2 元素的 tuple

**Schema 校验**（配置了 `schema_registry` 且事件类型已注册时启用）：
- 检查 `data` 中必填字段是否存在
- 检查字段类型是否匹配（支持 `(int, float)` 联合类型）
- 未注册的事件类型静默跳过

## 事件存储

### 内存层（热数据）

活跃窗口内的事件全部在内存中。`_event_log` 只追加不修改。

**实体索引**：`dict[str, list[Event]]`，entity_id → 事件列表（按时间有序）。查询使用 `bisect` 直接在 Event 对象上定位，O(log E + K)。

**空间索引**：`dict[tuple[int,int], set[Event]]`，chunk_key → 事件集合。按时间戳过滤。

### 归档层（冷数据）

通过 `configure(archive_path="save/events.db")` 或构造参数启用 SQLite 归档。

**存储**：三张表 — `events`（事件体）、`event_entities`（实体索引）、`event_edges`（因果边）。trim 时事件体 + 实体引用 + 边关系一同写入。`INSERT OR IGNORE` 保证幂等。

**查询**：查询方法自动合并内存和归档结果——锁内查内存（捕获 `earliest_ts`），锁外查 SQLite（`start_time` 到 `earliest_ts - 0.001`），两段不重叠，无重复无遗漏。

### 事件图 + 预热

`EventGraph` 维护邻接表（正向 `from_id → [(to_id, rel)]` + 反向 `to_id → [(from_id, rel)]`）。trim 时会移除被归档节点，但边已写入 `event_edges` 表。

`warmup_graph(max_events)` 启动时从归档边表批量恢复最近事件的边到内存，避免重启后因果追溯频繁走 SQLite 懒加载。

被移除节点的边仍可通过 `get_causal_chain(lookup=...)` 按需从归档恢复。

## 查询 API

| 方法 | 说明 |
|------|------|
| `get_events_in_range(start, end, *, type?, initiator?)` | 时间范围查询，自动合并归档 |
| `get_events_in_region(center, radius, *, start?, end?)` | 空间区域查询（chunk 粒度） |
| `get_entity_events(entity_id, start, end)` | 实体时间线查询 |
| `get_event_by_id(event_id)` | O(1) 内存 → 归档点查 |
| `event_count` | 内存中事件数 |
| `subscriber_count` | 活跃订阅数 |
| `stats` | 10 项运行指标 dict |

## 事件图 API

`world_tree.graph` 返回 `EventGraph`：

| 方法 | 说明 |
|------|------|
| `get_causal_chain(id, max_depth=10)` | 沿 caused_by 向上追溯 |
| `get_consequences(id)` | 直接后果事件 |
| `get_observers(id)` | 观测事件列表 |
| `has_path(from, to, max_depth=20)` | BFS 路径检测 |
| `warmup(edges)` | 批量加载边（从归档恢复） |
| `node_count` | 图节点数 |

## 生命周期

| 方法 | 说明 |
|------|------|
| `configure(*, archive_path?, max_memory_events?)` | 运行期配置归档和内存上限 |
| `warmup_graph(max_events=10000)` | 从归档边表预热因果图到内存 |
| `await_async()` | 等待所有异步回调完成（退出前调用） |
| `clear()` | 清空全部状态（仅测试） |

### trim（自动）

publish 时自动触发：`len > max_memory_events` → trim 到 `max//2`。

- **权重分层**：weight=1 首次即移除，weight=3 存活 2 次 trim，weight=5 存活 4 次
- **增量清理**：只移除被归档事件在索引中的条目，O(T)，T=移除数
- **边表写入**：trim 时边关系写入 `event_edges`，供 warmup 恢复

## 使用方式

```python
from ascend.world_tree import world_tree, Event, AffectedParty

# 配置（GameEngine.start() 中调用）
world_tree.configure(archive_path="save/events.db", max_memory_events=100_000)
world_tree.warmup_graph(max_events=10_000)

# 发布
world_tree.publish(Event(
    timestamp=game_time,
    location=(chunk_x, chunk_y, None, None),
    initiator_type="system",
    initiator_id="weather_system",
    affected=[AffectedParty("world", "subject")],
    event_type="weather_change",
    data={"from": "clear", "to": "rain"},
))

# 订阅
unsub = world_tree.subscribe("weather_change", handle_weather)
# 异步订阅（I/O 密集场景）
world_tree.subscribe_async("weather_change", handle_weather_slow)

# 查询
events = world_tree.get_entity_events("npc_1", start_time, end_time)
chain = world_tree.graph.get_causal_chain(event_id)

# 注册 schema（各模块在模块级调用）
world_tree.register_event_schema(
    "weather_change",
    required={"from": str, "to": str},
    optional={"intensity": float},
)

# 统计
stats = world_tree.stats
```

## 观测事件的感知判定

感知判定是轻量的物理计算，不涉及 LLM：

```
can_perceive(observer, event):
  if distance(observer.position, event.location) > observer.perception_range:
    return false
  if is_occluded(observer.position, event.location, tile_map):
    return false
  if observer.state in {sleeping, unconscious, blind, deaf}:
    return false
  return true
```

通过判定 → 生成 observation 事件，引用原事件 `observes`。

## NPC 时间线索引（待实现）

每个 NPC 维护一个事件引用列表，指向世界事件日志。是索引而非副本——共享事件只有一个存储位置，所有参与者通过索引指向它。

索引结构：`{event_id, relevance_score, memory_depth}`

- `relevance_score`：对 NPC 的重要性，由注意层计算
- `memory_depth`：随世界时间衰减，深度低于阈值的事件从活跃索引中移除（归档而非删除）

当前通过 `get_entity_events()` 直接查询实体参与的所有事件，NPC 时间线索引待心智系统实现后补充。

## 事件升级

个人事件 → 聚落事件 → 世界事件的升级判定不在世界树内部，而由相应的上层系统（群体社会、心智系统）判定后，生成新的事件，通过 `caused_by` 引用原始事件。

世界树只提供查询支持——"这个个人事件当前影响了多少人？"——供上层做升级判定。

## 快进推演循环

```
while world_time < target_time:
    step_size = determine_step_size()  // 根据当前局势动态调整
    events = []
    for system in active_systems:
        events += system.tick(step_size)  // 各系统产出事件
    for event in events:
        affected_npcs = get_affected_npcs(event)
        for npc in affected_npcs:
            if npc.needs_llm_judgment(event):   // 因果分叉点检测
                decision = llm_judge(npc, event, context)
                events += behavior.execute(npc, decision)
        world_tree.publish(event)          // 世界树负责存储和路由
        update_world_state(event)
    world_time += step_size
```

各系统的 `tick()` 返回事件列表，世界树负责存储和路由。本身不生成事件。

## 记忆衰减

```
memory_depth(t) = initial_depth × e^(-λ × t)
```

- `initial_depth` 由注意层关注程度决定
- `λ`（衰减速率）由事件类型决定：日常琐事 λ 大，情感重大事件 λ 小
- 低于阈值的记录从活跃索引移除，进入归档区
- 归档数据保留但不参与 LLM prompt 上下文构建
