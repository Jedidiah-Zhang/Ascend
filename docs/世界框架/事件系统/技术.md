# 世界树 — 技术

世界树（`ascend/world_tree/`）是连接所有模块的骨干。模块 A 发布事件时不关心模块 B 是否在监听，订阅者按 event_type 匹配接收。

## 模块结构

```
ascend/world_tree/
├── __init__.py          # 模块级单例 world_tree，公共 API 导出
├── tree.py              # WorldTree 类（发布/订阅/查询/trim）
├── event.py             # Event dataclass
├── affected.py          # AffectedParty dataclass
├── graph.py             # EventGraph 事件关系有向图
├── archive.py           # EventArchive SQLite 归档
└── schema.sqlite.sql    # 归档 DDL
```

## 事件数据结构

```
Event {
  id: UUID hex (自动生成),
  timestamp: float (游戏秒),
  location: (chunk_x, chunk_y, tile_x?, tile_y?),

  initiator_type: "system" | "npc" | "player",
  initiator_id: string,
  affected: [AffectedParty],     // {entity_id, role: "subject"|"witness"|"recipient"}

  event_type: string,            // "weather_change" "npc_action" "observation" ...
  data: dict[str, Any],          // 事件类型特定的内容，JSON 可序列化

  // 事件图关系
  caused_by: [EventID],          // 因果上游
  observes: EventID | None,      // 观测引用（仅观测事件）
  co_participants: [EntityID],   // 共享参与方
}
```

## 事件类型注册

每个产生事件的系统注册自己的事件类型 schema：
- 气候系统注册：`weather_change`（参数变更）、`weather_extreme`（极端天气）
- 行为系统注册：`npc_action`、`npc_speech`、`npc_decision`
- 生态系统注册：`population_change`、`bloom`、`migration`
- 感知系统注册：`observation`
- 实体系统注册：`entity_spawned`、`entity_despawned`、`entity_moved`
- 时间系统注册：`game_minute`、`time_skip`
- 日历系统注册：`day_end`、`day_change`、`hour_change`

## 世界树核心设计

### WorldTree 类

`WorldTree` 是事件系统的核心类。模块级单例 `world_tree` 通过 `from ascend.world_tree import world_tree` 获取。

```
WorldTree
├── _event_log: list[Event]             热数据（内存）
├── _entity_index: dict[str, list[int]]  实体索引（二分查找优化）
├── _spatial_index: dict[tuple, set[int]] 空间索引（chunk 分桶）
├── _graph: EventGraph                   因果有向图（所有事件的 ID + 关系边）
├── _archive: EventArchive | None        冷数据（SQLite，可选）
├── _lock: threading.RLock               线程安全
└── _validate: bool                      事件校验开关
```

### 发布流程

```
publish(event):
    锁外: _validate_event(event)         校验必填字段
    锁内: append → _event_log            O(1)
          更新 _entity_index             O(A) A=受影响方数
          更新 _spatial_index            O(1) set.add
          _graph.add_event               O(R) R=关系边数
          抓取订阅者快照                   O(1)
    锁外: _dispatch(event, callbacks)    回调 try/except 隔离
```

锁内操作 <1μs，回调在锁外执行，不阻塞后续 publish。

### 线程安全

- 所有公共读写方法由 `threading.RLock` 保护
- `publish()` 在锁内完成写入和订阅者快照，锁外执行回调
- 回调异常被 `try/except` 隔离，单个回调失败不影响其他订阅者
- 查询方法（`get_events_in_range` 等）持锁读取，确保一致性
- 归档查询在锁外执行（SQLite 有独立的并发控制），不占总线锁

### 事件校验

`WorldTree(validate=True)` 默认开启，在 publish 前校验：

- `event_type` 非空白
- `initiator_id` 非空白
- `initiator_type` ∈ {"system", "npc", "player"}
- `timestamp` 非负
- `location` 为至少 2 元素的 tuple

校验在锁外执行，快速失败。可通过 `WorldTree(validate=False)` 关闭。

## 事件存储

### 内存层（热数据）

活跃窗口内的事件全部在内存中。`_event_log` 只追加不修改。

**实体索引**：`dict[str, list[int]]`，entity_id → 日志索引列表。列表按时间有序（事件按时间顺序追加）。查询使用 Python `bisect` 模块二分定位，O(log E + K)。

**空间索引**：`dict[tuple[int,int], set[int]]`，chunk_key → 日志索引集合。查询时预计算时间对应的日志索引区间，用整数比较替代浮点时间戳比较。

### 归档层（冷数据）

`WorldTree(archive_path="save/events.db")` 启用 SQLite 归档。

**写入**：`trim()` 时自动将被移除的事件批量写入 SQLite，然后截断内存日志。幂等（`INSERT OR IGNORE`）。

**查询**：查询方法自动合并内存和归档结果——锁内查内存，锁外查 SQLite，按时间顺序合并返回。对调用方完全透明。

**不启用归档时**（`archive_path=None`，默认）：`trim()` 直接丢弃旧事件，行为不记录到磁盘。

### 事件图

`EventGraph` 始终在内存中，存储所有事件的 ID 和关系边（仅字符串引用，内存占用极小）。`trim()` 不动图结构——因果链永久可追溯。

因果链追溯只返回 ID 列表，不依赖事件体是否在内存中。

## 查询 API

| 方法 | 复杂度 | 说明 |
|------|--------|------|
| `get_events_in_range(start, end, *, event_type?, initiator_type?)` | O(log N + K) | 二分定位时间边界，返回按时间排序 |
| `get_events_in_region(center, radius, *, start?, end?)` | O(r² × D) | 整数预过滤时间范围，返回区域事件 |
| `get_entity_events(entity_id, start, end)` | O(log E + K) | 二分定位实体时间线，返回按时间排序 |

若启用归档且查询范围超出内存窗口，自动从 SQLite 合并归档结果（额外 +1-5ms 磁盘 I/O）。

## 事件图查询

`world_tree.graph` 返回内部 `EventGraph`，支持：

- `get_causal_chain(event_id, max_depth=10)` → 沿 `caused_by` 向上追溯因果链
- `get_consequences(event_id)` → 直接后果事件
- `get_observers(physical_event_id)` → 观测了某物理事件的 observation 事件
- `has_path(from_id, to_id, max_depth=20)` → BFS 检测有向路径
- `get_related(event_id)` → 与该事件有直接关系边的所有事件

### 事件图关系类型

| 关系 | 自动解析 | 说明 |
|------|----------|------|
| `caused_by` | 是 | 从 `event.caused_by` 列表自动建边 |
| `observes` | 是 | 从 `event.observes` 字段自动建边 |
| `co_participant` | 是 | 从 `event.co_participants` 列表自动建边 |
| `informed_by` | 否 | 需通过 `add_edge()` 手动追加 |

## 生命周期管理

### trim(before_time)

将早于指定时间的事件体从内存移除，重建实体和空间索引。若启用归档则先写入 SQLite。事件图不动——因果链完整保留。

调用时机：每游戏天一次，或存档时。O(N) 重建索引，N=剩余事件数。持锁期间阻塞 publish（通常 <10ms）。

### clear()

清空所有事件、索引、订阅和图结构。仅测试重置使用。

## 性能基准

10 万事件压力下（Python 3.14）：

| 操作 | 耗时 | 吞吐量 |
|------|------|--------|
| `publish` | 2μs | ~50 万次/秒 |
| `get_entity_events` | 1.6μs | 二分定位 O(log E) |
| `get_events_in_region` | 116μs | 整数预过滤 |
| `get_events_in_range` (2万条) | 477μs | 批量构建结果 |
| `trim` (1万条) | ~5ms | 锁内重建索引 |

## 使用方式

```python
from ascend.world_tree import world_tree, Event, AffectedParty

# 发布
world_tree.publish(Event(
    timestamp=game_time,
    location=(chunk_x, chunk_y, None, None),
    initiator_type="system",
    initiator_id="weather_system",
    affected=[AffectedParty("world", "subject")],
    event_type="weather_change",
    data={"from": "clear", "to": "rain"},
))

# 订阅
unsub = world_tree.subscribe("weather_change", handle_weather)

# 查询
events = world_tree.get_entity_events("npc_1", start_time, end_time)
chain = world_tree.graph.get_causal_chain(event_id)

# 启用归档
world_tree = WorldTree(archive_path="save/events.db")
world_tree.trim(before_time)  # 自动归档到 SQLite
```

## 观测事件的感知判定

感知判定是轻量的物理计算，不涉及 LLM：

```
can_perceive(observer, event):
  if distance(observer.position, event.location) > observer.perception_range:
    return false
  if is_occluded(observer.position, event.location, tile_map):
    return false
  if observer.state in {sleeping, unconscious, blind, deaf}:
    return false
  return true
```

通过判定 → 生成 observation 事件，引用原事件 `observes`。

## NPC 时间线索引（待实现）

每个 NPC 维护一个事件引用列表，指向世界事件日志。是索引而非副本——共享事件只有一个存储位置，所有参与者通过索引指向它。

索引结构：`{event_id, relevance_score, memory_depth}`

- `relevance_score`：对 NPC 的重要性，由注意层计算
- `memory_depth`：随世界时间衰减，深度低于阈值的事件从活跃索引中移除（归档而非删除）

当前通过 `get_entity_events()` 直接查询实体参与的所有事件，NPC 时间线索引待心智系统实现后补充。

## 事件升级

个人事件 → 聚落事件 → 世界事件的升级判定不在世界树内部，而由相应的上层系统（群体社会、心智系统）判定后，生成新的事件，通过 `caused_by` 引用原始事件。

世界树只提供查询支持——"这个个人事件当前影响了多少人？"——供上层做升级判定。

## 快进推演循环

```
while world_time < target_time:
    step_size = determine_step_size()  // 根据当前局势动态调整
    events = []
    for system in active_systems:
        events += system.tick(step_size)  // 各系统产出事件
    for event in events:
        affected_npcs = get_affected_npcs(event)
        for npc in affected_npcs:
            if npc.needs_llm_judgment(event):   // 因果分叉点检测
                decision = llm_judge(npc, event, context)
                events += behavior.execute(npc, decision)
        world_tree.publish(event)          // 世界树负责存储和路由
        update_world_state(event)
    world_time += step_size
```

各系统的 `tick()` 返回事件列表，世界树负责存储和路由。本身不生成事件。

## 记忆衰减

```
memory_depth(t) = initial_depth × e^(-λ × t)
```

- `initial_depth` 由注意层关注程度决定
- `λ`（衰减速率）由事件类型决定：日常琐事 λ 大，情感重大事件 λ 小
- 低于阈值的记录从活跃索引移除，进入归档区
- 归档数据保留但不参与 LLM prompt 上下文构建
