# 实体系统 — 技术

后端 `ascend/entity/`，前端 `scripts/world/main_world.gd`（直接处理 entity_snapshot/player_state 响应与 entity_born/died/moved 事件）。

## 数据结构（entity.py）

```python
class EntityType(IntEnum):     # 存在形态,按"是什么"分类
    CREATURE = 0
    PLANT = 1
    ITEM = 2
    STRUCTURE = 3

class Controller(IntEnum):     # 决策控制者,按"被谁控制"分类
    NONE = 0                   # 被动实体(物品/建筑/植物)
    AI = 1                     # NPC 心智系统/状态机
    PLAYER = 2                 # 玩家输入(意识转移即改写此标记)

@dataclass(slots=True)
class Entity:
    entity_type: EntityType
    chunk_x: int; chunk_y: int
    tile_x: int | None; tile_y: int | None
    born_at: int               # 诞生时的游戏时间
    layer_id: int = 0          # Z 层(0=地表,负数=地下),与渲染海拔正交
    controller: Controller = Controller.NONE
    id: str                    # UUID hex
    data: dict | None = None   # 组件容器,懒分配
```

- `__slots__` 消除 per-instance `__dict__`，`data` 为 None 时零开销
- **序列化约定**：跨进程/持久化一律用 `.name` 字符串，禁止用数值——枚举成员可能增删重排，数值会在版本间静默错位
- `data` 约定键：`fx`/`fy` = 精确 float 全局 tile 坐标（可移动实体）
- `global_xy` 属性：优先返回 `fx`/`fy`，否则由 chunk/tile 推导；与模块级 `split_coords(x, y)` 互为合成/拆分（floor 语义，负坐标 tile 恒在 `[0, TILE_MAP_SIZE)`）

## EntityManager（manager.py）

三索引 + 生灭/移动 API，所有变更发布世界树事件：

```
_entities: dict[id, Entity]                                  主表
_type_index: dict[EntityType, set[id]]                       类型索引
_spatial_index: dict[(layer,cx,cy,sub_x,sub_y), set[id]]     空间索引(16×16 tile sub-cell)
```

| API | 事件 | 说明 |
|---|---|---|
| `birth(type, cx, cy, tx, ty, *, layer_id, controller, data, game_time)` | `entity_born` | 事件 data 含 entity_id/entity_type/controller/position/layer_id/x/y（x/y 为全局 float 坐标） |
| `death(id, *, game_time)` | `entity_died` | 仅用于世界内生灭；停服清理直接丢弃管理器，不得借用本方法伪造历史 |
| `move(id, cx, cy, tx, ty, *, game_time)` | `entity_moved` | 同层移动；跨层是离散动作，留给专用 transition API（Issue #27） |
| `get(id)` / `by_type(t)` / `in_region(chunk, r, layer_id, center_tile, sub_radius)` / `all_entities()` | — | 查询；`all_entities` 供状态通道（快照/存档） |

索引维护：`_discard_from_index()` 在移除后**删除空桶**——空间索引键空间无上限（层 × chunk × sub-cell），空集残留在高周转下是无界内存泄漏。

## PlayerService（player.py）

玩家 = controller=PLAYER 的 CREATURE，本服务封装"玩家控制的那个实体"的生命周期与位置写入：

- `birth()`：出生点诞生，幂等
- `move_to(x, y)`：前端上报入口，壳子阶段无条件接受（未来在此加碰撞/速度校验）
- `teleport(x, y)` / `teleport_home()`：服务端主导传送，发布 `player_teleported` 事件供前端吸附
- 位置写入：精确值存 `data["fx"/"fy"]`，仅跨整数 tile 时调 `manager.move()`（避免高频移动刷世界树）

## 网络协议

### 状态通道（request-response）

```
entity_snapshot 请求 → {payload: {entities: [
    {id, entity_type, controller, x, y, layer_id}, ...]}}
    前端接入/读档后全量初始化;不产生历史

player_state 请求 → {payload: {entity_id, x, y}}
    告知前端"你控制的实体";位置为便捷冗余

player_move 请求 {payload: {x, y}} → {payload: {x, y}}
    上报本地位置,返回裁决后的权威位置
```

### 因果通道（EventBridge 广播）

`entity_born` / `entity_died` / `entity_moved` 事件经 EventBridge 转发前端。玩家实体的诞生早于任何客户端连接——前端不依赖历史事件，初始化一律走 snapshot。

## 前端（Godot）

`main_world.gd` 直接处理实体同步：

- **全量初始化**：连接后请求 `entity_snapshot` + `player_state`，创建对应的 Sprite3D billboard 占位节点
- **增量维护**：监听 `entity_born` / `entity_died` / `entity_moved` 事件，动态增删移实体视图
- **本地玩家**：标记 `entity_id` 为本地控制实体，忽略 `entity_moved` 回声，以本地输入预测为准

## 终端调试指令

`entity list` / `entity birth <type> [x y]` / `entity death <id前缀>`——见 [终端指令表](../../表现层/界面交互/终端指令.md)。death 拒绝移除 controller=PLAYER 的实体（PlayerService 会持悬垂引用，玩家死亡应走死亡机制）。
