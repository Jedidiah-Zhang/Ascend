# 实体系统 — 设计

实体是游戏世界中一切**值得个体化的存在物**：生物、植物、掉落物、建筑。实体系统定义它们的数据形态、生灭语义与前后端同步方式。

## 核心原则

### 玩家是 NPC 的一员

实体不因被玩家控制而特殊。玩家和 NPC 在世界中是同一种存在（CREATURE），唯一的区别是**决策模块**：NPC 的决策来自 AI（心智系统），玩家的决策来自键盘输入。"被谁控制"是实体上可替换的 `controller` 标记（NONE / AI / PLAYER），**不是**实体类型。

这是意识转移玩法的架构基础——玩家意识搬进另一个身体，本质只是改写两个实体的 controller 标记。也因此，实体类型枚举中没有也不应有 "PLAYER"。

### 组合优于继承

实体 = ID + 位置 + 组件集合。没有 `Deer extends Animal extends Creature` 式的类型继承树——因为基因可以增减身体部位、改变能力（见 基因系统），同一物种的个体在结构上就可能不同，继承树无法表达。

当前组件以 `data` 字典的约定键形式存在（如 `fx`/`fy` 精确坐标）；genome / body / needs / inventory 等组件 schema 待上游系统定型后规范化（Issue #24）。

### 两条通道

- **因果通道**（世界树 → EventBridge → 前端 / 未来 Perception → NPC）：只记录世界**内**发生的事。实体的诞生、死亡、移动是历史事实，进因果图，可被 NPC 感知。
- **状态通道**（request-response）：世界**外**的元操作——前端接入时拉取实体快照、存档读档、状态查询。不产生历史、不进因果图。

推论：**停服不是世界内事件**。停服时直接释放内存，不发布死亡事件——否则会向因果历史写入虚假死亡。实体状态的保存是存档系统（世界外）的职责。

### birth/death 与 spawn/despawn 的语义区分

| 术语 | 语义 | 归属 |
|---|---|---|
| **birth / death** | 实体在虚拟世界的生灭 | 世界内因果事件，走世界树（`entity_born` / `entity_died`） |
| **spawn / despawn** | 实体进入/离开前端渲染视野 | 表现层概念，不进世界树 |

现阶段视野 = 全部：前端收到 birth 即创建视图（spawn）、收到 death 即移除视图（despawn）。将来引入视野剔除与模拟保真度分层（视野外降级为概率模拟，Issue #26）后，despawn 不再等同 death——实体离开视野或降级时视图移除，但它没有死。

### 实体粒度

只有值得个体化的东西才是实体：

- 一棵可砍伐、有生长状态的树 = 实体；一片草地 = tile 属性
- 地上的掉落物 = 实体；容器/背包内的物品 = inventory 里的记录，不占实体表

粒度决策在物品/生态系统设计时逐类落实。

## 生灭的触发

当前只有两个生灭入口：玩家实体的诞生（引擎启动时）与终端调试命令（`entity birth/death`）。

生灭**规则**——谁在什么条件下诞生/死亡（生态刷新、繁殖、饥饿死亡、掉落物生成）——依赖生态、物品、经济系统的设计，规划为统一的规则层（Issue #22），各系统不直接调用底层生灭 API，保证每次生灭都有因果链可溯。

## 与其他模块的接口

- → **世界树**：生灭/移动发布为因果事件；未来感知模块（Issue #23）从这里取 NPC 的感知输入
- ← **基因系统**：基因决定实体的组件构成（Issue #24），是"组合优于继承"的根本原因
- ← **生态系统**：群系决定生物分布，生灭规则层是其执行入口（Issue #22）
- → **表现层**：EventBridge 广播生灭事件，前端实体视图（main_world.gd）增量维护渲染
- ← **存档系统**：实体表的持久化与恢复走状态通道（Issue #25），读档不发布诞生事件
- ← **心智系统**：NPC 的决策模块挂载为 controller=AI 实体的行为驱动（Issue #21 tick 入口）
