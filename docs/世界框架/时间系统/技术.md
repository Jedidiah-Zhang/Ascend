# 时间系统 — 技术

## 时钟实现

以 **tick** 为原子时间单位，时钟频率 **24Hz**（1 真实秒 = 24 tick）。不再以真实秒为基准计算游戏秒，所有时间状态（`clock.time`）均为整数 tick 计数。

### 时间常量（`constants.py`）

```python
GAME_MINUTE = 120        # 1 游戏分钟 = 120 tick（5 真实秒）
GAME_HOUR   = 7200       # 1 游戏小时 = 7200 tick
GAME_DAY    = 172800     # 1 游戏天 = 172800 tick（2 真实小时）
GAME_YEAR   = 62208000   # 1 游戏年 = 360 天
```

## WorldClock API

```python
from ascend.time import WorldClock, GAME_DAY

clock = WorldClock()          # 默认 epoch = 6 * GAME_HOUR（上午 6 点）
clock.time                    # int,   当前世界时间（tick）
clock.speed                   # float, 时间倍率（≥0，1=正常）
clock.paused                  # bool,  是否暂停（只读）
clock.tick_count              # int,   累计时间推进操作次数（见说明）

clock.game_days()             # float, 已经过的游戏天数
clock.game_years()            # float, 已经过的游戏年数
```

> **`tick_count` 语义**：这是"推进操作次数"，不是"世界实际经过的 tick 数"。`speed=2` 时一次 `tick()` 推进 2 tick 但 `tick_count` 只 +1。要获取世界实际经过的 tick 数请用 `time` 属性。

### 信号分发：直接回调，不经过 WorldTree

高频 tick 信号通过直接回调分发，避免每帧淹没 WorldTree。日历等订阅者检测到分钟/小时/天边界后，再自行发布语义事件到 WorldTree。

```python
unsub_tick = clock.on_tick(lambda game_time: ...)
unsub_skip = clock.on_skip(lambda skipped, game_time: ...)

unsub_tick()   # 取消订阅，幂等
unsub_skip()
```

- `on_tick(cb)`：每 tick 调用 `cb(game_time: int)`，返回 unsubscribe 函数
- `on_skip(cb)`：跳转时调用 `cb(skipped: int, game_time: int)`，返回 unsubscribe 函数

### tick() — 每帧推进

```python
clock.tick()  # 由 GameEngine 每帧调用，无参数
```

- 暂停或 speed≤0 时不推进
- 使用浮点累加器：每帧累加 `speed`，整数部分为推进 tick 数
- 每帧最多触发 1 次 on_tick 回调（无论推进多少 tick）
- 例：speed=0.5 时每 2 帧推进 1 tick

### run_to(target) — 批量模拟

```python
clock.speed = 120              # 设置高速
clock.run_to(clock.time + 3 * GAME_DAY)  # 模拟 3 天
```

- 以当前 speed 逐 tick 推进到目标时间
- 期间正常触发 on_tick 回调，日历自动检测日/时边界
- 完成后恢复原始暂停状态
- target 必须在当前时间之后，否则抛出 ValueError
- 暂停状态请用 `resume()` 或 `step()`，瞬跳请用 `skip()`

### skip(ticks) — 瞬间跳转

```python
clock.skip(30 * GAME_DAY)      # 瞬间跳 30 天，不模拟中间过程
```

- 直接增加时间，触发 1 次 on_skip 回调（携带 skipped、game_time）
- 日历注册 on_skip 后自行检测日/时边界并补发 `day_change`、`hour_change` 等
- ticks 必须 > 0

### step() — 调试单步

```python
clock.step()                   # 强制 1 tick，忽略暂停和 speed
```

- 调试/终端手动控制用
- 正常游戏循环不应调用

## GameCalendar

日历不主动推进时间——它**注册** WorldClock 的 `on_tick`/`on_skip` 回调，根据 `game_time` 检测分钟/小时/日变更，并在边界跨越时发布语义事件到 WorldTree。

```python
from ascend.time import WorldClock, GameCalendar

clock = WorldClock()
calendar = GameCalendar(clock=clock, start_day=1)
calendar.day                  # int, 当前游戏日（从 1 开始）
calendar.hour                 # int, 当前小时（0-23），初始化前返回 0
calendar.minute               # int, 当前分钟（0-59），初始化前返回 0
calendar.elapsed_days         # int, 从起始日至今经过的天数
calendar.day_change_count     # int, 累计日期变更次数
calendar.hour_change_count    # int, 累计整点变更次数

calendar.day_at(game_time)       # 计算指定时间对应的游戏日
calendar.time_of_day(game_time)  # 当天内的 tick 偏移 [0, GAME_DAY)
calendar.shutdown()              # 取消订阅，释放资源
```

### 事件发布顺序

日期跨天时发布顺序：`day_end` → `day_change`，确保日终结算先于新日初始化。

整点变更发布顺序：`hour_change` 在日变更检测之后，确保 hour 已基于最新 day 计算。

分钟变更：`minute_change` 在 hour_change 之后检测。

## 事件体系

### day_end

每日结束时发布（day_change 之前），用于日终结算：

```python
{
    "day": int,            # 即将结束的日
    "elapsed_days": int,   # 截至该日的经过天数
}
```

### day_change

日期变更时发布，触发群体/生态等日更模块：

```python
{
    "day": int,                # 新日
    "previous_day": int,       # 上一日
    "elapsed_days": int,       # 从起始日至今经过的天数
    "day_change_count": int,   # 累计日期变更次数
    "skipped_days": int,       # 跳过的天数（skip 跨多日时 > 0）
}
```

### hour_change

整点变更时发布，用于高频定期任务：

```python
{
    "day": int,                 # 当前日
    "hour": int,                # 新小时
    "previous_hour": int,       # 上一小时
    "hour_change_count": int,   # 累计整点变更次数
}
```

### minute_change

分钟变更时发布，用于需要分钟级更新的模块（如天气引擎）：

```python
{
    "day": int,
    "hour": int,
    "minute": int,
    "game_time": int,
}
```

## 性能

- **实时 tick**：24Hz 累加器，每帧运算为一次浮点加法 + 一次整数截断 + 一次回调遍历，开销极低
- **run_to 模拟**：speed=120 时 3 游戏天约 4320 次 tick，纯事件驱动，毫秒级完成
- **skip 跳转**：O(1)，仅一次回调（日历的日/时边界补发为恒定次数，与跳转跨度无关）
- **LLM 调用**：快进期间仅于因果链分叉点触发，日常行为全自动
- 具体 LLM 调度策略留待 MVP 后细化

## 整合点

### WorldClock — 依赖

WorldClock 是纯计时器，除 `ascend.log` 和 `ascend.config` 外无其他模块依赖（不 import WorldTree、EntityManager 等运行时服务）。通过 `on_tick` / `on_skip` 回调列表向订阅者分发高频 tick 信号，不经过 WorldTree。

### WorldClock — 被依赖

| 调用方 | 方式 | 调用的方法 / 属性 |
|--------|------|------------------|
| **GameEngine** (`game.py`) | 直接构造，持有引用 | `WorldClock()`、`tick()`（每帧）、`pause()`/`resume()`、`paused`、`time` |
| **GameCalendar** (`time/calendar.py`) | 构造注入 | `clock.on_tick(_on_tick_advance)`、`clock.on_skip(_on_skip_advance)` |
| **WeatherEngine** (`weather/weather_engine.py`) | 构造注入 | `clock.time`（读取当前 tick 数，计算季节、昼夜、大气漂移） |
| **CommandExecutor** (`terminal/executor.py`) | 构造注入 | `clock.time`、`clock.speed`、`clock.paused`、`clock.step()`、`clock.skip()` |
| **PlayerService** (`entity/player.py`) | 构造注入 | `clock.time`（记录操作时间戳） |

### GameCalendar — 依赖

| 依赖 | 方式 | 用途 |
|------|------|------|
| `WorldClock` | 构造注入 | 注册 `on_tick` / `on_skip` 回调，接收时间推进信号 |
| `WorldTree` | 模块级单例 import | 发布 `minute_change`、`hour_change`、`day_end`、`day_change` 四个语义事件 |

### GameCalendar — 被依赖

| 调用方 | 方式 | 调用的方法 / 属性 |
|--------|------|------------------|
| **GameEngine** (`game.py`) | 直接构造，持有引用 | `GameCalendar(clock=...)`、`shutdown()` |
| **CommandExecutor** (`terminal/executor.py`) | 构造注入 | `calendar.day`、`calendar.hour`、`calendar.minute`、`calendar.elapsed_days` |

### 间接消费（通过 WorldTree 事件）

- **WeatherEngine**：订阅 `minute_change`，作为唯一 tick 来源驱动天气计算
- **EventBridge**：通配符订阅 `*`，转发全部日历事件到 Godot 前端
- **未来模块**（群体/生态/心智）：订阅 `day_change` 做日更结算
