# 时间系统 — 设计

## 基底时间

**2 真实小时 = 1 游戏天**。不赶人，也不慢到育种跑不动。

比传统殖民模拟游戏（20分钟/天）慢 6 倍，让日常行为（吃喝、对话、制作）有自然的呼吸感，不被高频打断。

以 **tick** 为原子时间单位，时钟频率 24Hz（1 真实秒 = 24 tick）。浮点累加器支持任意小数速度倍率。

| 常量 | tick 数 | 对应真实时间 |
|------|---------|------------|
| 1 游戏分钟 | 120 tick | 5 真实秒 |
| 1 游戏小时 | 7200 tick | 5 真实分钟 |
| 1 游戏天 | 172800 tick | 2 真实小时 |
| 1 游戏年（360 天） | 62208000 tick | 720 真实小时 |

## 时间速度控制

放弃固定的"时间模式"枚举，改为**连续速度倍率 + 暂停状态**。无论日常、睡眠、快速旅行还是长跳闭关，本质都是同一套模拟引擎以不同速度运行。

### 速度倍率（speed）

- `clock.speed`：浮点倍率，≥0，每次可设任意值
- `speed = 1.0`：正常速度，每帧推进约 1 tick
- `speed = 0.5`：半速，每 2 帧推进 1 tick
- `speed = 120`：高速，每帧跳 1 游戏分钟
- `speed = 0`：等效暂停，tick() 不推进

### 预设场景（概念对应）

| 概念 | speed | 用途 |
|------|-------|------|
| 正常 | 1.0 | 实时游戏，逐帧渲染 |
| 暂停 | 0 或 pause() | 菜单/对话/规划时冻结世界 |
| 睡眠 | 120 | 跳过 8 游戏小时（`run_to` 高速模拟） |
| 快速旅行 | 120+ | 跳过数周到数月（`run_to` 高速模拟） |
| 长跳闭关 | 极大 | 跳过数年到数十年，skip() 瞬间跳转 |

### 暂停

- `clock.pause()` / `clock.resume()`：显式暂停/恢复
- `clock.paused`：只读状态
- 暂停时 tick() 空转不推进，但 step() 仍可强制单步（调试用）

### 推进方式

1. **逐帧 tick**：`clock.tick()` — GameEngine 每帧调用，按当前 speed 推进。累加器处理小数速度，不够 1 tick 时不推进。触发 `on_tick` 回调。

2. **批量模拟 run_to**：`clock.run_to(target)` — 以当前 speed 逐 tick 推进到目标时间。期间正常触发 `on_tick` 回调，日历据此检测并发布 `day_change`、`hour_change` 等语义事件。

3. **瞬间跳转 skip**：`clock.skip(ticks)` — 不模拟中间过程，直接跳 N tick。触发一次 `on_skip` 回调，日历据此检测日/时边界并补发事件。

4. **强制单步 step**：`clock.step()` — 精确 1 tick，忽略暂停和 speed，仅调试用。

### 事件体系

高频 tick 信号通过 WorldClock 直接回调（`on_tick`/`on_skip`）分发，不经过 WorldTree——避免每帧淹没事件总线。GameCalendar 注册这些回调，检测到分钟/小时/天边界后再发布语义事件到 WorldTree。

| 事件 | 发布者 | 频率 | 数据 |
|------|--------|------|------|
| `minute_change` | GameCalendar | 跨分钟时 | day, hour, minute, game_time |
| `hour_change` | GameCalendar | 跨整点时 | day, hour, previous_hour, hour_change_count |
| `day_end` | GameCalendar | day_change 前 | day, elapsed_days |
| `day_change` | GameCalendar | 跨天时 | day, previous_day, elapsed_days, day_change_count, skipped_days |

GameCalendar 注册 `clock.on_tick` / `clock.on_skip` 回调，自动检测分钟/时/日边界并发布相应事件，无需上游手动计算。

## 寿命梯度

| | 凡人种 | 长生种 | 修仙者 |
|---|---|---|---|
| 寿命 | 几十年 | 数百年 | 理论上无限（随境界延长） |
| 生育率 | 正常 | 低 | 极低或不育 |
| 分布 | 主要人口 | 聚居特定聚落 | 少量散布于智慧生物中 |
| 角色 | 育种代际来源 | 社会记忆锚点 | 玩家建立深度关系的跨时代同伴 |

### 修仙者境界

修仙者只修长生，不修武力。境界越高寿命越长。境界划分待细化——但早期只需 2-3 层：筑基（寿 200）、金丹（寿 500）、元婴（寿 1000+）。

玩家作为修仙者，在漫长寿命中见证凡人代际更替、社会演化兴衰。

## 快进模拟

### 原则：同一套模拟引擎，不同速度

快进不是另一套系统——是模拟引擎以更高 speed 跑完（或 skip 跳转），不渲染、不逐帧展开。NPC 依然用自己的技能、朝着自己的目标前进，撞上其他 NPC 时自然产生共享事件，规模够大时自然升级为世界事件。

没有"结婚概率表"——是 NPC A 和 NPC B 各自追求目标的过程中产生交集，关系值到位、人格匹配，自然走到一起。

### 生成顺序

```
1. 外部世界事件（天灾等——不需要 NPC 参与，先确定主干）
2. NPC 个人轨迹（沿时间线推进，能力+目标+人格驱动）
3. 共享事件同步（一方产生 → 自动写入所有参与方）
4. 个人事件升级（超过显著性门槛的纳入世界事件表）
5. 受影响 NPC 以新世界事件为参照重新调整
```

### 共享事件同步

结婚、决斗、交易等涉及多方的共享事件，一方生成时自动同步到所有参与方的时间线。同一条数据，各自关系页面上有各自视角的描述。

冲突解决：若目标 NPC 的时间线上已存在冲突事件（如 B 第 6 年要远行，A 要跟 B 结婚），由人格参数决定结果——可能一方让步、双方妥协、或产生撕裂事件（"婚礼上逃婚"）。

### LLM 调用

快进期间，NPC 一生中只有少数节点真正需要 LLM 决策——重大人生抉择、关系破裂、生命威胁——在因果链分叉点、不做决策就无法继续推演该 NPC 后续轨迹时唤醒。

几十年跨度内，一个聚落只有部分 NPC 会触发 LLM 调用。具体调度的优先级和上限留待 MVP 后根据实测确定。

### 回来后

一切 LLM 对话需要的历史上下文——世界事件、人生轨迹、共享事件的参与方视角——都以结构化摘要形式存储。玩家找任何 NPC 对话时，LLM 直接检索相关摘要，开销和平时对话一致。不会因为跳了 50 年而每个 NPC 都唤醒一次 LLM。

## 与其他模块的接口

- → **生命个体**：时间驱动生理需求衰减；长跳后 NPC 在时间线上的健康变化由生理系统判定
- → **基因系统**：长跳是育种循环的时间基础——凡人几代更替后可见性状遗传结果
- → **心智系统**：快进后 NPC 的人生摘要作为 LLM 上下文；待回顾决策点触发 LLM 唤醒
- → **群体社会**：关系图谱随时间演变；世界事件作为所有 NPC 共享的记忆参照
- → **世界框架/事件系统**：世界事件是快进模拟的生成基础；快进期间模拟的小粒度事件可能升级为新的世界事件
