# 界面交互 — 技术

## 模块结构

```
scripts/
├── config.gd                  集中配置（RefCounted，常量与后端 config.py 同步）
├── autoload/
│   ├── connection.gd          Connection 单例（autoload）
│   └── msgpack.gd             MsgPack 静态编解码（当前 JSON）
├── world/main_world.gd        MainWorld：输入分发 + 消息路由 + 调试数据推送
└── ui/
    ├── terminal_widget.gd     TerminalWidget (Control)
    ├── debug_overlay.gd       DebugOverlay (Control)
    ├── debug_section.gd       DebugSection 基类 (RefCounted)
    ├── debug_sections/*.gd    9 个分区子类
    └── event_log.gd           EventLog (Control)
```

场景分层（main.tscn）：`World`（游戏世界）→ `TerminalLayer`（CanvasLayer, layer=100）→ `DebugLayer`（CanvasLayer, layer=200）。

## Connection 单例

管理与 Python 后端的 TCP 连接，`_process` 每帧轮询驱动状态机。

### 协议

- 帧格式：4 字节大端长度前缀 + 消息体（JSON，`MsgPack.encode/decode` 封装，切 MessagePack 时调用方无需修改）
- 消息：`{type, seq, payload, ...}`；`type ∈ {request, response, event, error}`，`seq` 自增
- 消息上限 16 MiB；信号：`connection_established / connection_lost / message_received`

### 后端进程自管理

启动流程（非阻塞状态机，避免主线程卡顿）：

1. `_start_backend()` 先非阻塞探测 `127.0.0.1:9081`（`PROBE_TIMEOUT = 0.2s`）
2. 端口已开放 → 直接连接（复用已运行后端）；未开放 → `OS.create_process` 拉起 `.venv/bin/python backend/run_server.py`
3. 等待期间每 0.5s 重新探测，超时 10s 报错并杀进程
4. 退出时（`NOTIFICATION_PREDELETE`）`OS.kill` 回收后端

断线后 2s 间隔自动重连。

### 后台解码线程

JSON 解析移出主线程：`_read_messages()` 主线程只做拆帧，帧体经 Mutex+Semaphore 队列交给 `Thread` 解码，`_collect_decoded()` 主线程收集结果并发射 `message_received`。`last_process_us` 记录每帧耗时供性能分区显示。

## MainWorld 消息路由

`_on_message()` 按 `type` 分发：

- `response`：`get_chunks` → MapDisplay；`entity_snapshot` → EntityLayer 全量对齐实体视图；`player_state` → EntityLayer 标记本地控制实体（`entity_id`）+ MapDisplay 初始位置吸附；`get_weather` → 天气分区；`terminal_cmd` → 终端输出
- `event`：`player_teleported`（MapDisplay 位置吸附 + 日志，先于 overlay 检查处理）、`entity_born / entity_died / entity_moved`（EntityLayer 增量维护实体视图 + 日志，先于 overlay 检查处理）、`minute_change`（时间分区 + TPS 推算 + 日期分隔线）、`temperature_change / humidity_change / wind_change / sunshine_change / precipitation_start / precipitation_stop`（气候/天气分区 + 事件日志，经 3×3 chunk 位置过滤 `_is_within_event_log_view`）
- `error`：`push_error`

实体视图同步（工厂在 `world/entity_layer.gd`，视图节点 `world/entity_view.gd`，详见世界框架/实体系统文档）：

- 连接建立时先请求 `entity_snapshot`（状态通道全量），再请求 `player_state`（标记本地控制的 entity_id）
- 此后靠 `entity_born/died/moved` 事件（因果通道）增量维护；本地控制实体忽略 `entity_moved` 回声
- Player 与其他实体走同一条 EntityView 渲染管线，仅位置驱动不同（本地=输入预测，其余=事件驱动）

玩家位置同步（权威实体在后端 `PlayerService`）：

- 连接建立时请求 `player_state` 初始化本地位置；出生数据晚到不覆盖（`_has_player_state` 标记）
- WASD 移动本地插值即时显示，MapDisplay 每 0.2s（`MOVE_REPORT_INTERVAL`）节流上报 `player_move`（仅位置变更时发送）
- 壳子阶段后端原样接受上报；未来加校验后 `player_move` 响应可作为纠正吸附点

调试数据推送（`_update_debug_sections()`，仅面板可见时执行）：

- 每帧推送相机/玩家/区块统计/性能计时
- 玩家移动到新 tile 时才查询地形/气候（避免每帧字典查找）
- 每 0.5s 轮询一次 `get_weather`（`_weather_query_accum` 限流）

## DebugSection 架构

分区基类为 RefCounted（非节点），DebugOverlay 统一 `_draw()` 渲染，避免节点树开销：

```
DebugSection (基类)
├── label: String            分区标题（DebugOverlay.get_section(label) 检索键）
├── enabled: bool             关闭时跳过渲染
├── get_lines() -> PackedStringArray   子类覆写，返回文本行
└── update_from_backend(data)          事件型分区覆写
```

三种数据更新模式：

- **自取型**：`get_lines()` 内直接读 Engine/Performance（内存、连接分区）
- **推送型**：MainWorld 每帧设置属性（相机、玩家、区块分区）
- **事件型**：`update_from_backend()` 注入后端数据（时间、气候、天气、地形分区）

DebugOverlay 每 0.25s（`REFRESH_INTERVAL`）`queue_redraw()`，绘制时先收集全部行测量背景宽度，再统一绘制。

## TerminalWidget

全部通过 `_draw()` 自绘（无 RichTextLabel 子节点）：

- 打开时 `_input()` 捕获全部键盘事件并 `set_input_as_handled()`，实现输入独占
- 输出环形缓冲 500 行、历史 100 条、单行输入上限 200 字符，光标 0.5s 闪烁
- 本地指令注册表：`register_command(name, handler, help_text)`（模仿 DebugSection 模式），handler 签名 `(args: PackedStringArray) -> String`；内置 `clear`，MainWorld 注册 `tp`
- 回车后匹配本地指令直接处理；`help`/`?` 显示本地帮助并继续转发后端追加远程帮助；其余发射 `remote_command_submitted` 信号由 MainWorld 转发后端
- 指令表见 [终端指令](./终端指令.md)

## EventLog

- `push_event(line)` 头插，超 20 条截断；`_pending_redraw` 标记 + 0.5s 节流合并高频事件的重绘
- 固定面板宽 280px，避免每帧测量字符串宽度

## 性能约定

- 调试 UI 一律自绘 + 节流刷新（0.25s/0.5s），关闭时零开销
- 覆盖层 `mouse_filter = IGNORE`，不参与命中测试
- 热路径缓存节点引用（`@onready`），禁止每帧 `get_node`
