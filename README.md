# Ascend

> 自下向上构造因果世界的 AI 原生模拟平台

[English](README.en.md) | 中文

---

## 项目定位

Ascend 兼具双重身份：

**研究平台：** 自下向上构造的、具备完整因果机制的程序化世界，以及覆盖全系统的结构化事件因果图（WorldTree）。研究目标是基于因果图与时空序列，训练针对该世界的世界模型——Ascend 为此提供可复现、可干预、可追溯的实验环境。

**游戏：** 玩家作为世界中的一名个体，通过基因改造影响族群演化方向。NPC 由 AI 驱动——对话、决策、记忆、关系全部动态生成，而非预设文本和行为树。

---

## 设计理念

### 世界先于智能体

世界先于智能体存在——地形、气候、水文、生态、时间流逝——这些构成 NPC 感知的输入信号。NPC 并非被置入静态场景，而是诞生于一个已独立演化的因果空间中。

### 物理演化驱动语义涌现

温度为连续物理量，降雪为语义阈值跨越的涌现现象。世界不预设语义态，仅由物理演化规则推导——生命周期作为因果链的涌现属性自然浮现。

### AI 为一等公民

所有游戏系统存在的首要意义，是为 AI NPC 提供结构化、可检索、可追溯的上下文。基因系统、事件总线、关系图谱、目标系统——每个模块最终都服务于同一个映射：**感知 → 决策**。游戏侧问「自主智能体需要怎样的世界感知，才能产生可信行为」；研究侧问「世界模型能否从因果图与时空序列学到该映射」——同源问题，共用同一数据底座。

---

## 研究动机

现有世界模型研究大多在因果结构不透明、不可干预的环境中训练，缺乏可追溯性。另一面，程序化生成、物理因果仿真、结构化事件日志等技术虽已成熟，却鲜有被有意识地面向「世界模型训练」进行系统性设计。

Ascend 试图填补该交叉领域的空白：以可复现、可干预的因果世界产出时空序列，训练世界模型；训练完成的模型反过来替换 NPC 的 LLM 决策层——形成「世界生成 → 数据产出 → 模型训练 → 反哺决策」的闭环。

## 系统架构

> 当前为早期阶段蓝图，各模块架构仍在演进。

### 技术栈

- **Godot 4.x**：渲染、UI、输入、音频
- **Python 后端**（`ascend-backend/`）：全部核心逻辑
- **通信**：MessagePack over TCP，localhost
- 依赖见 `requirements.txt`

### 架构分层

系统由三部分组成：

```
① 世界生成           ② NPC 决策            ③ 行为接口
   ↓ 感知输入            ↓ 动作输出
   因果图（WorldTree） ← 动作执行回写世界
```

**① 世界生成层** — 程序化生成的因果空间，可独立运行与演化。向 NPC 提供感知接口（数值量 → 语义标签）作为决策输入。状态变更经事件因果图（WorldTree）记录，既支撑 NPC 感知，亦构成世界模型的时空序列训练样本。

**② NPC 决策层** — 感知世界，结合人格参数与记忆，输出动作。LLM 显著性评分驱动按需唤醒；设计文档中的五层架构（角色 / 目标 / 动机 / 技能 / 执行）属游戏侧细节，见 `docs/心智系统/`。世界模型基于因果图与时空序列离线训练，训练完成后可替换 LLM 决策层。

**③ 行为接口层** — 定义 NPC 可选行为的契约。决策层从行为集中选择，经接口执行并回写世界，世界层据此继续演化。

### 目录结构

```
ascend-backend/   Python 后端（核心逻辑）
ascend-frontend/  Godot 前端
docs/             设计文档
tests/            测试
lang/             多语言资源
requirements.txt  Python 依赖
```

## 快速开始

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cd ascend-backend
python run_server.py
```

Godot 前端位于 `ascend-frontend/`，需 Godot 4.x 打开。
